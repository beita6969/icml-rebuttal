**W1：算子库可扩展性与功能重叠工具**（新增：Tables R2–R4 | 论文：Prop. 1, 4）

---

感谢这个问题，我们希望澄清 FlowSteer 设计中的一个关键区别。FlowSteer 工作在*元层面（meta-level）*：它并非在工具集中搜索要调用的工具，而是通过多轮编辑动作（add/delete/modify/set_prompt/控制结构）构造 workflow graph（Definition 1）。12 个算子是用于 workflow 构建的可组合构件，而非待搜索的工具清单。这与 tool-search 系统存在本质区别——后者在扩展到数百个 API 时会面临搜索空间和歧义消解的挑战。

具体而言，分解式动作空间（Section 4.1，Proposition 1）将动作类型与算子解耦——新增算子仅带来线性增长 $O(\\lvert\\mathcal{A}\_{\\text{type}}\\rvert + \\lvert\\mathcal{O}\\rvert)$。Flow-Director 通过语义描述（Table 7）而非索引选择算子，因此功能重叠由策略对算子语义的理解来消解，而非穷举搜索。

迁移实验（新**表 R2–R4**）提供了实证支持：移除算子导致针对性退化（−Verify/Test：APPS −14.05，DS-1000 −20.31；−ScEnsemble/Aggregate：AIME −6.67），验证各组的独特作用；用未见实现替换后性能几乎无损（均值 IID +0.3 / OOD +0.1）；新增全新算子则选择性提升目标任务（+Search：TriviaQA/NQ +5.47/+8.59 EM；+Debugger：APPS/DS-1000 +2.35/+3.91），非目标任务零退化。

我们认同扩展到更大规模算子库是重要的未来方向，一个自然的扩展是分层算子选择（类别 → 算子），我们的分解式动作空间天然支持这一点。同时，当前 12 个算子已提供广泛的任务覆盖：Proposition 4（附录 B.2）证明它们通过满射映射实现了全部 7 个认知原语（生成、验证、聚合、条件分支、顺序链接、集成、格式化），理论最低需 5 个算子——我们的算子库超出此下界，且上述迁移实验证实无冗余。

**W2：计算成本与训练-推理权衡**（新增：Tables R6, R7 | 论文：Appendix G, Figures 4–5）

---

新**表 R7** 提供了不同训练范式的单步成本对比（均在 2×A100 80GB 上，批量 = 36 个问题）：

| 指标 | SFT | GRPO | FlowSteer |
|---|---|---|---|
| 单步时间 | 5.35s | 1,257s | 1,004s |
| 补全 Tokens | 0 | 108,315 | 132,146 |
| 训练 Tokens | 5,534 | 6,917 | 36,828 |
| **总 Tokens** | **5,534** | **115,232** | **168,974** |

尽管存在多轮 Canvas 交互，FlowSteer 的单步耗时（1,004s）实际上比单轮 GRPO（1,257s）**低 20%**，这得益于向量化 rollout 优化（附录 G）：32 路并发轨迹、批量 API 调用、缓存工作流状态、提前结束跳过。FlowSteer 因多轮反馈产生更多总 tokens（168,974 vs. 115,232），但并行化执行使其每步更高效。总训练在 ~300 步内收敛（~8 GPU 小时），且训练一次即可零样本部署到 6 个甚至更多架构各异的后端（Figure 4），无需逐后端调优。

推理端，FlowSteer *更高效*：Figure 5(a-b) 显示 CWRPO（Full）的 token 消耗和交互轮数低于所有消融变体——RL 教会策略何时停止。新**表 R6** 确认任务自适应成本：GSM8K 平均 8.3 轮 / \$0.0012，AIME 12.4 轮 / \$0.0019。
